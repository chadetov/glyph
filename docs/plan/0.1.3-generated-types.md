# 0.1.3 — Generated types, not hand-written DTOs

Target: kill the "manual DTO per API" tax for backend/API work (surfaced by
Ashfaq, and the same root as the #1 interop weakness) **without bending the
manifesto**.

## The principle (north star)

> We don't infer or alias types — we **generate** them. Every DTO is a real,
> checked-in, descriptor-bearing `type` declaration, whether a human or a
> generator wrote it.

The manifesto is against types that *lie* (implicit / unvalidated) and symbols
that *scatter* — not against generation. `z.infer` is objectionable because the
type is implicit (no greppable declaration, no runtime descriptor). A **generated**
`type X = {...}` is arguably *more* manifesto-compliant than a hand-written one: a
generator can't typo a field, and the output still greps to one place, still
carries a D8 runtime descriptor, still diffs cleanly.

## Guardrails (apply to every task)

- **Generation is an external tool** (like `glyph fmt` / `glyph regen`), never a
  language primitive. No new type operators (`z.infer`-style), no revival of the
  abandoned annotation-heavy direction, no relaxed strictness.
- **Every generated/materialized type is a real `type X = {...}`** — greppable,
  one place — with a D8 descriptor, so boundary data is validated.
- **Output is canonical** (`glyph fmt`); regeneration produces clean, minimal
  diffs. Stamp generated files with `@generate by:` (Q40 documentation metadata)
  so `glyph regen` can refresh them.
- Advances **Q40** (type-driven generation) and **Q41** (FFI ergonomics),
  resolving them toward "generate real code," not language magic.

## Track A — `type` is the zod replacement  ·  ship first, low effort, ~zero risk

Glyph already does **type → validator** (declare `type NewTask = {...}`, the
emitter generates `T.schema`, and `json.parse<NewTask>(body)` validates for free).
That is the inverse of zod's **value → type** and the manifesto-native version of
it. Most of Ashfaq's pain is removed just by making this obvious.

- **A1.** `docs/guide/typed-apis.md`: "You don't need zod for your own DTOs —
  declare a type, get validation." Show `json.parse<T>`, `T.is`, `T.parse`, and
  the request-body → typed-DTO flow over `std/http`.
- **A2.** A worked backend example `examples/05_rest_api.glyph`: `std/http` server,
  typed request/response DTOs, boundary validation, an auth-middleware handler
  wrapper. (There is no backend example today — this is also the "show a real app"
  ask from Pelin/Sunil.)
- **A3.** Audit `std/json` + `std/http` for parse/validate rough edges; add at most
  a small convenience (e.g. a one-call "validate a `Request` body into `T`" helper).
  No language change.

**Done when:** writing one `type` gives a validated boundary, and a runnable REST
example ships.

## Track B — `glyph gen openapi`  ·  the headline feature, medium effort

A spec → committed Glyph. Highest leverage for backend teams and the cleanest fit
with Q40 (external CLI, real code out).

- **B1.** New CLI subcommand family: `glyph gen openapi <spec.{json,yaml}> --out <dir>`.
- **B2.** Map OpenAPI 3 / JSON Schema → Glyph: objects → records, `enum`/`oneOf` →
  tagged unions, `nullable`/optional → `Option`/optional fields, arrays →
  `Array<T>`, `$ref` → named types.
- **B3.** Emit typed **client functions** per operation and/or **handler**
  signatures (errors-as-values over `std/http`).
- **B4.** Emit canonical Glyph; idempotent regeneration; stamp `@generate by:
  openapi <spec>` so `glyph regen` refreshes it.
- **B5.** Descriptors come free → request/response validation straight from the
  spec.

**Done when:** point at an OpenAPI spec, get typed + validated DTOs and a
client/handlers you can grep and edit.
**Risk:** OpenAPI has many corners. MVP = the common 80% (objects, enums, `$ref`,
arrays, nullable); emit a clear diagnostic and skip the rest rather than guess.

## Track C — materialize external types  ·  hardest, medium-high, stretch

Turn an external TS/zod type from an ambient phantom (type-checks but no
descriptor, not greppable) into a first-class Glyph type you own.

- **C1.** `glyph gen dts <file.d.ts> --out` (and/or `gen zod`): read a TS type /
  zod schema, emit an equivalent Glyph `type` + descriptor.
- **C2.** MVP shapes: object types, unions, arrays, primitives, optional fields.
  Advanced TS (conditional / mapped / deep generics) → clear diagnostic + skip.
- **C3.** Implementation: a small node helper using the TS compiler API to read the
  type, feeding a Glyph type-emitter. Self-contained; no new language surface.
- **C4.** Keep the ambient `.types/` path for un-materializable surface —
  documented as **type-only** (no runtime validation), which is the boundary the
  manifesto draws anyway.

**Done when:** `glyph gen dts user.d.ts` yields a real, validated Glyph `type`.
**Risk:** TS type parsing is genuinely hard. Scope tightly; if B slips, move C to
0.1.4.

## The `glyph gen` surface

One family, consistent with `regen`: `glyph gen <openapi|dts|zod> <src> --out <dir>
[--client] [--handlers]`. All output is normal, committed, greppable Glyph.

## Scope: all three ship in 0.1.3 (decided)

A, B, and C all ship in 0.1.3. Build order is A → B → C (fastest and
lowest-risk first), each landing behind its own tests + a worked example, so the
version accretes value even while C (the hardest, TS-type parsing) is in flight.

- **A** — docs + REST example. Near-zero risk; removes most of the pain immediately.
- **B** — `glyph gen openapi`. The headline; the real build.
- **C** — the TS/zod materializer. Hardest (TS compiler-API integration); MVP shapes
  first (objects/unions/arrays/primitives/optional), clear diagnostics for the rest.
  It ships, but scoped to the common cases — advanced TS is a documented follow-up,
  not a blocker.
