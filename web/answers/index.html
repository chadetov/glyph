Questions engineers actually asked

The honest answers — what works, and what's still on the way.

We put Glyph in front of working engineers — frontend, backend, and skeptics of the whole premise — and let them build real things. Every question they raised is here, grouped and answered straight. No question is softened; where something isn't ready, it says so.

Interop & the existing JS/TS world

The most-asked cluster: can Glyph live inside the code and libraries I already have?

Can I import my existing TypeScript, JavaScript, and Node libraries?Yes

Yes. Glyph compiles to ordinary TypeScript and rides npm: a .glyph module emits a .ts file that imports from any npm package and is itself importable from your existing .ts. That is the whole point of targeting TypeScript instead of shipping a new runtime. Package specifiers with hyphens and npm scopes (react-hook-form, @hookform/resolvers/zod) now parse directly — a first-contact gap an engineer hit, since fixed. Interop crosses the TypeScript boundary today; a dedicated @ffi declaration syntax is planned for v2, but you don't need it to use the npm ecosystem now.

What about React libraries like react-hook-form or zod?Partly — open

This is the sharpest friction an engineer found, and we won't pretend otherwise. Some React-library patterns collide with Glyph's deliberately strict grammar: JSX prop spreading ({...register()}) and types derived from a runtime value (z.infer<typeof schema>) both need a thin TypeScript adapter today, rather than working inline. The easy parts already got easier — hyphenated/scoped imports and aria-*/data-* attributes now work — but the per-library adapter tax is real. Making a complex-typed TS library usable without a bespoke adapter is an active, top-priority open question. See the hardening page and the open questions.

Do custom hooks and React Context actually work?Yes — Context improving

This was one engineer's single biggest worry, and it turned out overstated. A custom hook is just a function that calls other hooks — no special machinery. React Context works today via createContext / useContext; member-expression element names (<ThemeContext.Provider>) are landing so the provider needs no workaround. And complex state is genuinely safer than TypeScript: a useReducer over a tagged union gets compile-time exhaustiveness — add an action and every reducer that misses it fails to compile.

Tooling & developer experience

“Doing it by hand felt like Notepad” was fair. Here's the editor and error story.

Is there a VS Code / editor experience?Yes

Yes. There is a language server and a VS Code extension: syntax highlighting, diagnostics with stable error codes, hover types, go-to-definition, completion, and format-on-save via glyph fmt. Rename and find-references land in v1.1. The editor setup guide covers VS Code and any other LSP editor.

How does debugging work?TS tools today

You debug the emitted TypeScript with the tools you already use — Node's inspector, browser devtools, your IDE's JS/TS debugger. The output is clean, readable TS that maps closely to your source, not minified soup. Source maps that step straight back to .glyph line numbers are on the roadmap, not in v1 yet.

Are the error messages any good?Elm-quality bar

That's an explicit goal, and engineers held us to it. Every diagnostic carries a stable code, a one-line fix, and often the design rationale; glyph --explain E0203 prints the long form. Where the highest-traffic mistakes still gave generic messages we fixed them specifically: writing if now says “Glyph has no if; match is the only conditional” instead of “unexpected token”. The hardening page lists more of those.

The language, deliberately

Some friction is on purpose. Here's why, and where we met engineers halfway.

No ternary, no if/else, <for> instead of .map?By design

Correct, and intentional. match is the only conditional, which means there is exactly one shape to read and grep for. These restrictions earn their keep against the verifiability and greppability pillars — the reasons Glyph exists — so the answer is better documentation and better error messages, not loosening the rule. An agent, the primary author, has no ten years of muscle memory to unlearn; it writes whatever the grammar says.

Why TypeScript, and not Go or a fresh language?Adoption

Because TypeScript is where agents already write the most code and where the ecosystem you depend on — npm, React, Node — lives. Glyph is adoptable one file at a time only because its target is TS; you can't drop Go into a TS codebase file by file. The point that Go's small keyword surface gives an agent fewer ways to go wrong is fair — it's exactly why Glyph keeps a small core — and Go is now in the density benchmark.

Can an AI even write a brand-new language?

The deepest skeptic's question — and the one we were most surprised by.

LLMs are trained on existing languages. Isn't a new one too hard for them?Held up

The fair worry: an LLM is a next-token predictor trained on the huge body of existing-language code, so a language with almost no training data should be hard to write. In practice it held up far better than expected, for a structural reason: Glyph is a disciplined subset of TypeScript with an ML core. Roughly 80% transfers from TS and functional-programming intuition on the first try; the restrictions are few and enumerable, and each fails fast at compile time. In our own adversarial testing, an agent built a real REST API — typed records, tagged-union validation, auth middleware — to a green build in about two iterations. It is “TypeScript with the footguns removed,” not an alien language.

Evidence & maturity

Where we have proof, where we have honest future work.

Does using fewer tokens actually mean an agent writes better code?No, and we don't claim it

No, not on its own. Density is the last of four pillars, not the first. The wedge is verifiability and greppability: code that can't lie about its types and symbols that grep to one place. What we have not yet measured is an agent success-rate or tries-to-correct number against TypeScript on the same task. That study is honest future work, and we say so on the benchmarks page.

Where are the real example apps — REST/GraphQL, auth, middleware?Growing

A real backend is expressible today, and cleanly: std/http ships an errors-as-values server (serve / Handler / Request / Response), auth is a plain higher-order function wrapping handlers, and validation is exhaustive match over tagged unions. The examples and corpus in the repo show working programs; more fully-built apps (auth + middleware, REST and GraphQL) are what we want early adopters to push on.

Are the libraries and community there yet?Thin, honestly

Honestly, the Glyph-native library surface and community are thin today — it's early. What softens it is that Glyph rides npm rather than replacing it, and adoption is incremental: one file at a time inside an existing TypeScript project, not an all-or-nothing rewrite.

The safety — what engineers liked

The part that landed. Here's what it actually buys you.

What does the safety concretely buy me over TypeScript?Real guarantees

Four things a silently-missed case can't survive: exhaustive match over sealed tagged unions (add a variant and every match that misses it fails to compile); Result plus the ? operator, so errors are values you must handle, not exceptions that vanish; @example tests that live at the point of definition and run at build time; and @redact for PII. These are the guarantees we then attack relentlessly — see how, and what that found, on the hardening page.
