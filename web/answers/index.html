Answers

The questions engineers actually ask.

Straight, specific answers — each one names the strength, shows a real example, and states the current edge honestly. Start wherever your doubt is. New to Glyph? The 10-minute Start Here is the fastest first run.

Start with the problem

01What stops an agent from silently breaking my code?

No any, no casts, exhaustive match, runtime-checked types.

 02Will agent edits stay clean, or reformat my whole file?

A one-line change is a one-line diff. Canonical format, no reflow.

 03Can an agent — or I — find where anything is defined?

One name, one declaration form. grep always lands on it.

Your stack & ecosystem

04Can I use my existing TypeScript, JS, and npm libraries?

Yes — import any npm package, and import Glyph back into .ts.

 05Do I have to hand-write a validator for every API?

No. Declare a type, get a validated boundary for free.

 06Can I generate a typed client and server from my API spec?

glyph gen openapi/zod/dts — committed, greppable code.

 07Does it work with React — components, hooks, and Context?

First-class JSX, hooks, and exhaustive reducer actions.

 08How do I share mutable state between functions?

A store: a const whose .set/.update are the write-surface.

Should I adopt it?

09Why TypeScript, and not Go or a new language?

Because that's where your ecosystem is, and it's why adoption is per-file.

 10Why not just strict TypeScript + zod + eslint + ts-pattern?

Non-optionality: the language the agent can't config-disable.

 11Is Glyph worth it if I'm not using AI?

Yes — the guarantees are plain engineering wins a human gets directly.

 12Can an AI reliably write a language it's barely seen?

~80% transfers from TS; the compiler's tight feedback closes the rest.

Living with it

13What's the editor, debugging, and error experience?

LSP + VS Code extension, and Elm-quality errors with stable codes.

 14When it breaks in production at 3am, what does on-call see?

Readable TS you own, with source maps back to your .glyph.
