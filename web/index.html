TypeScript your agents can't quietly break.

Glyph compiles to clean TypeScript and reads almost identically — so you adopt it one file at a time. But its types can't be lied to, every symbol greps to one place, and a one-line change stays a one-line diff. The footguns agents fall into are gone by construction.

The tax you already pay

If you build with AI agents on a TypeScript codebase, you know the daily cost. Glyph is TypeScript with these footguns removed.

  • ✗ Agents hallucinate APIs, because TypeScript erases its types and runtime no longer agrees with compile time.
  • ✗ Agents paper over uncertainty with any and as unknown as T, and the compiler lets them.
  • ✗ You ask for a one-line change and get a whole file rewritten, because the formatter cascades on every edit.
  • ✗ You grep for a symbol and find ten unrelated matches — overloads, decorators, merged namespaces.
  • ✗ Reviewers can't tell if a PR is correct, because the diff is twelve hundred lines of reflowed whitespace.

Glyph removes each one — by construction.

Why Glyph wins

In order of what matters most. The first two are the wedge — problems you feel every day. The last two are the polish that makes it pleasant.

◇
01 · The wedge

Verifiability

What the types say is true at runtime. No any, no casts, no erasure — every type keeps a runtime descriptor, and match must cover every case. The bugs an agent ships that tsc --strict waves through don't compile here.

⌕
02 · The wedge

Greppability

One name, one place. Every symbol has exactly one syntactic form at its declaration — no overloads, decorators, implicit this, or namespace merging. grep "fn parseUser" finds the definition. Always.

≡
03 · The polish

Diff stability

A one-line change is a one-line diff. Fixed formatting, one element per line, trailing commas, no reflow, no barrel files. Agents stop rewriting whole files; reviewers read the change, not the churn.

⊟
04 · The polish

Abstraction

Say what you mean: pattern matching over switch ladders, Result over thrown exceptions, named records over tuples — a small orthogonal core, not TypeScript's accreted layers.

Measured, not asserted

The case is structural and reproducible. Small sample, stated plainly — the full methodology and caveats are one click away.

Verifiability — the bugs that cost you
2 of 2 caught at compile time

Two mistakes agents make constantly — a missing match variant and an unsafe cast — are rejected by Glyph and accepted by tsc --strict. Wrong code that compiles is the expensive failure. Glyph won't let it.

Greppability — one name, one place
1 match, every time

Every symbol has exactly one declaration form — no overloads, decorators, or merged namespaces to scatter it.

Diff stability — reviewable by default
1 line → 1 line

A one-line Glyph edit produces a one-line TypeScript diff through the transpiler, and glyph fmt adds no churn. The diff is the change, not the noise around it.

Questions engineers actually ask

The honest answers — what works today, and what is still on the way.

Building with an AI agent — where does it start?

Point the agent at glyphlang.io/llms.txt (also committed as AGENTS.md in the repo). It is one document that takes an agent from zero to correct, runnable Glyph: install and run commands, the program shape, a full syntax cheat-sheet, every standard-library signature, the import model, and the gotchas — no source-diving. With the toolchain installed, glyph llms prints the same thing offline, and glyph init scaffolds a runnable starter project.

Can I import my existing TypeScript, JavaScript, or Node libraries?

Yes. Glyph compiles to ordinary TypeScript and rides npm: a .glyph module emits a .ts file that imports from any npm package and is itself importable from your existing .ts. That is the whole point of compiling to TS instead of shipping a new runtime. Today interop crosses that TypeScript boundary; a dedicated @ffi declaration syntax is planned for v2, but you do not need it to use the npm ecosystem now. The external-imports guide shows the import rule, the .types/ mechanism, and a worked example.

How does debugging work?

You debug the emitted TypeScript with the tools you already use — Node's inspector, browser devtools, your IDE's JS/TS debugger. The output is clean, readable TS that maps closely to your source, not minified soup. Source maps that step straight back to .glyph line numbers are not in v1 yet; they are on the roadmap. Until then, the generated .ts is meant to be read and stepped through directly.

Does it work with React, Node, and frameworks like NestJS?

React is built in: component declarations compile to React function components and JSX lowers to React.createElement. Anything that runs on Node runs Glyph's output, so Express, Nest, and the rest consume Glyph-generated TypeScript like any other module. For a plain Node backend, std/http now ships a small errors-as-values server (serve / Handler / Request / Response). Deep, decorator-heavy framework integration at scale is still maturing — that is the kind of thing we want early adopters to push on.

Is there a VS Code / editor experience?

Yes. There is a language server and a VS Code extension: syntax highlighting, diagnostics with stable error codes, hover types, go-to-definition, completion, and format-on-save via glyph fmt. Rename and find-references are landing in v1.1. The editor setup guide walks through installing it in VS Code (and any other LSP editor), format-on-save, and troubleshooting.

Why TypeScript, and not Go or a fresh language?

Because TypeScript is where agents already write the most code and where the ecosystem you depend on — npm, React, Node — lives. Glyph is adoptable one file at a time only because its target is TS; you cannot drop Go into an existing TS codebase file by file. The point that Go's small, stable keyword surface gives an agent fewer ways to go wrong is a fair one — it is exactly why Glyph keeps a deliberately small core — and Go is now included in the density benchmark above.

Does using fewer tokens actually mean an agent writes better code?

No, not on its own — and we do not claim it does. Density is the last of four pillars, not the first. The wedge is verifiability and greppability: code that cannot lie about its types and symbols that grep to one place. What we have not yet measured is an agent success-rate or tries-to-correct number against TypeScript on the same task. That study is honest future work, and we say so on the benchmarks page.

How do I build a real app with it?

Run glyph init to scaffold a runnable project, then read the five-minute tour and the worked example programs (a CLI tool, a React component, and more), or experiment in the playground. For a backend, std/http gives you an HTTP server today. Larger end-to-end starters — a full service with auth and middleware — are still in progress, because more than one engineer asked for exactly that.
